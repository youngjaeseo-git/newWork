## 2. 작업을 시작하기 전 (Before starting)

- 코드를 바꾸기 전에 관련 파일·맥락을 먼저 파악한다.
- 요구사항이 모호하면 임의로 정하지 말고 선택지를 제시한다.
- 큰 작업은 먼저 계획을 세우고 합의한 뒤 진행한다.

## 3. 프로젝트 진행 관리 (Project Tracking)

> 모든 프로젝트는 **시작 단계에서 아래 추적 파일들을 먼저 만들고**, 진행하면서 계속 갱신한다.
> 역할이 겹치지 않게 파일을 분리한다.

### 시작 단계 (킥오프) — 순서 중요

1. **브레인스토밍 먼저** — 사용자가 생각·아이디어·요구사항을 두서 없이 쏟아내면, Claude는 그 내용을 그대로 `brainstorming.md` 에 기록한다. 정리하거나 판단하지 않는다. 일단 다 받는다.
2. **경쟁 제품 피드백 리서치** — 비슷한 제품·서비스가 있으면, 실제 사용자 피드백을 모아 `개선 요청 Top 5` 와 `불편/불만 Top 5` 를 정리한다. (아래 "경쟁 제품 피드백 리서치 루틴" 참고)
3. **계획 수립** — `brainstorming.md` 와 리서치 결과를 바탕으로 전체 계획과 세부 주간 계획을 작성해 `plan.md` 에 저장한다.
4. **추적 파일 생성** — `todo.md`, `work.md`, `history.md`, `docs/` 를 함께 만든다.

> 브레인스토밍 단계에서 사용자에게 먼저 물어볼 것:
> "어떤 걸 만들고 싶으세요? 지금 떠오르는 것 아무거나 말씀해 주세요. 정리 안 해도 됩니다."

### 경쟁 제품 피드백 리서치 루틴
> 비슷한 제품·서비스가 있을 때, 그쪽 사용자들이 이미 겪은 불편과 원하는 개선을
> 미리 흡수해서 우리 프로젝트 방향에 반영하기 위한 루틴.

- **언제** — 프로젝트 시작 시 1회(권장). 이후 주요 마일스톤마다 선택적으로 다시 돌린다.
- **어디서** — 커뮤니티(레딧, 네이버 카페/블로그, 관련 포럼 등), 플레이스토어·앱스토어 리뷰, 제품 후기. (도구: `/deep-research`)
- **무엇을** — 가장 큰 것 5개씩 두 묶음으로 정리한다.
  - **개선 요청 Top 5** — 사용자들이 "이게 있었으면" 하는 것
  - **불편/불만 Top 5** — 사용자들이 가장 답답해하는 것
  - 각 항목에 **출처 링크**와 **언급 빈도/강도**를 함께 적는다.
- **출력** — `docs/YYYY-MM-DD_경쟁제품-피드백.md` 로 저장한다.
- **적용 체크 (중요)** — 정리한 10개 항목을 표로 만들어 사용자에게 보여주고,
  각 항목을 **우리 프로젝트에 적용할지(✅ / ❌ / 보류)** 를 직접 체크받는다.
  임의로 반영하지 않는다.
- **반영** — 적용하기로 한 항목만 `plan.md`(범위/마일스톤)와 `work.md`(할 일)에 옮긴다.


### 추적 파일 구조
| 파일 | 역할 | 갱신 주기 |
|---|---|---|
| `brainstorming.md` | 초기 아이디어·요구사항 원본 덤프 (수정하지 않음) | 프로젝트 시작 시 1회 |
| `plan.md` | 전체 계획 + 주간 세부 계획 + 현재 상태 요약 | 계획 변경 시 / 주간 |
| `todo.md` | 진행 상태 **대시보드** — 계획 대비 빠른지/늦은지, 현재 어느 구간(phase)인지 | 수시 |
| `work.md` | 할 일 / 한 일 체크리스트 (실제 작업 단위) | 작업할 때마다 |
| `history.md` | 매주 한 일을 간략히 기록 (주간 아카이브) | 주 1회 |
| `docs/` | 진행 중 발생하는 산출물 보관 | 산출물 생길 때마다 |

### 각 파일 규칙
- **brainstorming.md** — 사용자 말을 가공 없이 기록. 나중에 plan.md 를 작성할 때 참고 원본이 된다. 내용을 덮어쓰지 않는다.
- **plan.md** — 프로젝트 시작 시 작성. `전체 목표 → 마일스톤 → 주간 계획` 순으로. 맨 위에 "현재 상태" 요약을 둔다. **확정된 결정은 "확정 결정" 절에 기록**해 같은 논의를 반복하지 않는다.
- **todo.md** — `계획 대비 +N일 빠름 / -N일 늦음`, `현재 구간: N주차 / 마일스톤 X` 처럼 진척 상태를 한눈에 보이게 한다. (거시 관점) 아직 정하지 못한 것은 **"결정 필요" 절에 옵션과 장단점을 적고 "보류"로 표시**한다 — 임의로 정하지 않는다.
- **work.md** — 지금 할 일과 끝낸 일을 체크박스(`- [ ]` / `- [x]`)로 관리하고, 진행하면서 바로바로 반영한다. (미시 관점)
- **history.md** — 매주 끝에 그 주에 한 일을 3~5줄로 요약해 누적한다. 과거 기록은 지우지 않는다.
- **docs/** — 산출물은 `docs/YYYY-MM-DD_제목.md` 형식으로 저장한다 (날짜를 앞에 두면 시간순 정렬됨).

### 공통 원칙
- 각 파일 상단에 **마지막 업데이트 날짜**를 적는다.
- 작업을 끝낼 때 관련 추적 파일을 갱신하는 것을 "완료"의 일부로 본다 (12번 체크리스트 참고).

## 5. 배포 & 수익 모델 (Deployment & Monetization)

> 프로젝트가 어느 정도 완성도(MVP)를 보이는 시점에 다룬다. 너무 일찍 정하지 않는다.
> 배포·수익 모델 모두 **Claude가 선택지·장단점을 정리해 제시하고, 사용자가 직접 결정**한다. 임의로 정하지 않는다.

### 배포 (Deployment)
- **언제** — 핵심 기능이 동작하고, 교차 검증·`/security-review` 를 통과한 뒤.
- **어디에 배포할지 결정** (제품 형태별):
  - 모바일 앱: **Google Play Store**, Apple App Store
  - 웹앱: Vercel / Netlify / Cloudflare Pages, 자체 서버(도커 등)
  - 데스크톱: 각 OS 스토어 / 직접 배포
- **Play Store 배포 준비물 체크리스트**:
  - Google Play 개발자 계정 (등록비 $25, 1회)
  - 앱 서명 키 (Play App Signing 권장)
  - 스토어 등록 자료: 아이콘·스크린샷·짧은/긴 설명·기능 그래픽
  - 개인정보 처리방침 URL (필수)
  - 콘텐츠 등급 설문, 타깃 연령, 데이터 보안(Data safety) 양식
  - 최신 타깃 API 레벨 충족
  - 내부 테스트 → 비공개 테스트 → 단계적 출시 순서 권장
- **출력** — 배포 계획·체크리스트는 `docs/YYYY-MM-DD_배포계획.md` 로 저장한다.

### 수익 모델 (Monetization)
- **언제** — MVP가 사용자에게 가치를 검증한 뒤 결정. 수익화는 UX와 트레이드오프가 크므로 신중히.
- **선택지와 장단점**:

| 모델 | 설명 | 장점 | 단점 |
|---|---|---|---|
| 광고 (AdMob 등) | 배너·전면·보상형 광고 | 무료 진입장벽, 초기 수익 | UX 저하, 단가 낮음 |
| 인앱 결제(소비성) | 코인·아이템 등 캐시 결제 | 게임·앱에 적합, 반복 매출 | 결제 거부감, 스토어 수수료 |
| 구독 | 월/연 정기 결제 | 안정적 반복 매출 | 지속 가치 제공 부담 |
| 유료 앱 | 다운로드 시 1회 결제 | 단순, 광고 없음 | 진입장벽 높음 |
| 프리미엄(Freemium) | 기본 무료 + 고급 기능 유료 | 넓은 사용자 + 전환 | 무료/유료 경계 설계 어려움 |

- **적용 체크 (중요)** — 위 선택지를 표로 보여주고, 어떤 모델을 쓸지 **사용자가 직접 선택**받는다. 임의로 정하지 않는다.
- **반영** — 결정한 모델을 `plan.md`(마일스톤)와 `work.md`(구현 작업)에 옮긴다.
- **주의** — 결제·광고는 각 스토어 정책과 관련 법(소비자보호·개인정보·환불 등)을 반드시 확인한다.

### 글로벌 런칭 고려 (Going Global)
> **불특정 다수를 상대로 수익화를 노리는 서비스**라면, 런칭 이전에 국내 전용으로 낼지
> 글로벌까지 노릴지를 한 번 점검한다. 시장이 클수록 수익 상한도 커지지만, 현지화·법·결제
> 부담이 함께 늘어난다. 나중에 끼워 넣으려면 비싸므로 **설계 단계에서 미리 결정**한다.

- **언제** — 수익 모델을 정하는 시점에 함께 점검한다. (코드가 굳기 전이 쌈)
- **먼저 물어볼 것** — "이 서비스, 국내만 볼까요 / 글로벌도 노릴까요? 노린다면 1차 타깃 시장(언어·국가)은 어디일까요?" → **사용자가 직접 선택**받는다. 임의로 정하지 않는다.
- **글로벌을 노린다면 점검 체크리스트**:
  - **현지화(i18n/L10n)**: 다국어 문자열 분리(하드코딩 금지), 통화·숫자·날짜/시간 형식, 시간대(UTC 저장), 필요 시 RTL.
  - **법·규제**: 개인정보(EU GDPR, 미국 CCPA, 한국 개인정보보호법), 쿠키/추적 동의, 연령 등급, 데이터 보관 위치(데이터 레지던시).
  - **결제·세금**: 글로벌 결제수단(스토어 인앱결제는 자동, 웹은 Stripe/PayPal 등), 통화·환율, 부가세/세금(EU VAT 등) 처리.
  - **인프라**: CDN, 리전/지연시간, 가까운 곳에서 서빙.
  - **스토어/유통**: 국가별 출시 설정, 각국 콘텐츠 등급, 스토어 등록 자료·키워드(ASO) 현지화.
  - **문화·UX**: 카피 톤, 색·이미지·아이콘이 특정 문화에서 문제되지 않는지.
  - **운영**: 고객 지원 언어·시간대.
- **최소 비용 전략 (실무)**: 처음부터 전부 다 하지 않는다. **① 코드는 다국어 가능하게 설계(문자열 분리)** 해두고, **② 출시는 국내 먼저 → 검증 후 시장 확장** 하는 단계적 접근이 보통 가장 싸다.
- **출력** — 시장 결정과 체크리스트는 `docs/YYYY-MM-DD_글로벌런칭-검토.md` 로 저장하고, 적용할 항목만 `plan.md`·`work.md` 로 옮긴다.

## 6. 일하는 방식 (How I work)

- 변경은 작고 검증 가능한 단위로 나눈다.
- 기존 코드의 스타일·네이밍·구조를 먼저 따른다 (새 패턴을 함부로 도입하지 않는다).
- 가능한 한 테스트로 검증한 뒤 "됐다"고 말한다. 안 됐으면 그대로 보고한다.
- **세션 시작 루틴**: 작업 전 `plan.md` 의 "현재 상태"와 `work.md` 의 미완료 항목을 먼저 확인하고, 이어서 할 일을 정한다.
- **막혔을 때 규칙**: 같은 방법으로 2번 시도해 안 되면 그대로 반복하지 않는다. 가정을 점검하고 접근을 바꾸거나, 막힌 지점·시도한 것·다음 선택지를 정리해 보고한다.
  - 시간을 많이 쓴 문제는 `docs/` 에 **트러블슈팅 로그**(증상 → 시도와 결과 표 → 확인된 사실 → 근본 원인 → 교훈)로 남긴다. (양식: `templates/troubleshooting.md`)
  - 한 번에 하나씩 좁혀간다. 전체를 한꺼번에 고치려 하지 않는다.
- **검증 근거**: "됐다"고 말할 때는 근거(테스트 결과·실행 로그·재현 절차)를 함께 제시한다. 추측으로 완료를 보고하지 않는다.
  - **빌드 ≠ 동작 검증**: 타입체크·빌드·문법 통과는 "문법 검증"일 뿐이다. 외부 연동·런타임 등 실제 동작은 별도로 확인해야 하며, 어디까지 검증됐는지(문법까지인지, 실제 동작까지인지)를 사용자에게 명확히 구분해 전달한다.
- **Data-First 원칙**: 외부 시스템(API, DB, 외부 데이터 소스 등)과 연동하는 기능은 **실제 데이터 구조를 먼저 확인한 뒤** 코드를 작성한다. 추측으로 짜지 않는다.
  - 응답의 실제 라벨·필드명·값 형식을 확인하고 나서 구현한다.
  - 확인용은 일회성 명령이 아니라 **스크립트/파일로 만들어** 재현 가능하게 한다.
- **받은 정보는 파일로 보존**: 사용자가 수동으로 전달한 데이터·확인 결과·결정은 파일(`docs/`, 추적 파일 등)에 저장하고, 같은 것을 다시 요청하지 않는다. 새 작업 전에 관련 파일이 이미 있는지 먼저 확인한다.
- **별도 에이전트 교차 검증**: 함수·기능을 구현하면, 구현한 작업자의 자가 점검으로 끝내지 않고 **별도 에이전트에게 검토·검증·테스트를 맡긴다.** 구현자는 자기 코드의 사각지대를 못 보기 때문이다.
  - 코드 품질·버그 검토: `/code-review` (또는 리뷰 전용 서브에이전트)
  - 실제 동작 검증: `/verify`
  - 검토자는 구현 의도가 아니라 **결과(동작·엣지케이스·테스트 통과 여부)** 를 기준으로 판단한다.
  - 검토에서 나온 지적은 무시하지 말고, 반영하거나 반영하지 않는 이유를 남긴다.
- **관측 가능성(Observability) — 로그·에러 기록**: 개발자가 모든 사용자 환경을 다 테스트할 수 없으므로, 기능을 만들 때 **운영 중 문제를 추적할 수 있는 로그/에러 리포팅을 함께 설계**한다.
  - 핵심 동작·실패 지점·예외에 로그를 남긴다 — **언제·어디서·무슨 입력으로** 실패했는지 알 수 있게.
  - 사용자 환경에서 발생한 오류는 수집·저장해 **재현과 근본 원인 파악**에 쓴다. 분석 결과는 `docs/` 트러블슈팅 로그로 이어진다.
  - 로그 레벨(debug/info/warn/error)을 구분하고, 운영에서는 과도한 로그로 노이즈를 만들지 않는다.
  - **로그에 비밀·개인정보(토큰, 비밀번호, 민감 개인정보)를 남기지 않는다.** (8번 "하지 말아야 할 것"과 연결)
- **치명적 에러·크래시 리포팅 (Crash Reporting)**: 프로그램이 죽거나 치명적 버그가 나면, 개발자가 **어디서 터졌는지 자동으로 전달받을 수 있게** 한다. 개발 환경의 소수 테스트로는 모든 사용자 환경·기기 차이를 못 잡기 때문이다.
  - **전역 에러 핸들러**로 처리되지 않은 예외·크래시를 붙잡는다. (예: JS `window.onerror`·`unhandledrejection`, 백엔드 uncaught exception/프로세스 종료 핸들러, 모바일 크래시 핸들러)
  - **개발자에게 도달하는 채널**로 보낸다. 규모에 따라 선택한다:
    - 가벼움: 에러를 로컬/원격 로그 파일에 적재 → 사용자가 "오류 신고"로 전송
    - 중간: 백엔드 엔드포인트(`POST /api/error-report`)로 수집·저장
    - 본격: 에러 리포팅 서비스(Sentry, Firebase Crashlytics, Rollbar 등) 연동 + 새 크래시 알림(메일/Slack)
  - **무엇을 담을까**: 발생 시각, 에러 메시지·스택, 앱/빌드 버전, OS·브라우저·기기, 직전 동작·화면(브레드크럼). **비밀·개인정보는 제외.**
  - **계획에 반영**: 불특정 다수 대상 서비스라면 크래시 리포팅을 처음부터 `plan.md` 에 넣는다 (나중에 붙이면 비싸다). 수집된 크래시는 `docs/` 트러블슈팅 로그로 이어 분석한다.
- **예외 처리·견고성 (Robustness)**: 잘못된 입력이나 비정상 상황에서도 프로그램이 안전하게 동작하도록 **예외 처리를 빠짐없이 적용**한다.
  - **입력 검증**: 외부·사용자 입력은 신뢰하지 않는다. 타입·범위·형식·null/빈 값을 **먼저 검증**한 뒤 사용한다.
  - **모든 분기 처리**: 조건문(`if`/`switch`)은 예상 못 한 경우(`else`/`default`)까지 처리한다. "그럴 리 없는" 경로를 비워두지 않는다.
  - **예외를 삼키지 않는다**: 빈 `catch` 금지. 예외는 로그를 남기고(6번 관측 가능성과 연결) 적절히 처리하거나 전파한다.
  - **검증 시 비정상 입력도 테스트**: 정상 입력뿐 아니라 잘못된 입력·경계값·빈 값·중복 등으로도 동작을 확인한다.
- 변경 후에는 포맷터/린트를 실행해 깨진 곳이 없는지 확인한다.

## 7. 커밋 / PR 규칙 (Commit & PR)

- 커밋 메시지는 명확하고 설명적으로 쓴다.
- 사용자가 요청할 때만 커밋/푸시한다 (멋대로 push 금지).
- PR은 기본적으로 draft로 만든다.
- **커밋 메시지는 Conventional Commits 규칙을 따른다.**
  - 형식: `<type>(<scope>): <설명>` — scope는 선택.
  - type: `feat`(기능), `fix`(버그), `docs`(문서), `style`(포맷), `refactor`(리팩토링), `test`(테스트), `chore`(잡일/설정), `perf`(성능), `build`, `ci`.
  - 제목은 명령형·현재형으로 짧게(50자 내외), 본문에 "왜"를 적는다.
  - 예) `feat(auth): 소셜 로그인 추가`, `fix(api): 빈 응답 처리 누락 수정`

## 8. 코드 스타일 / 품질 원칙 (Code style)

> 주 언어/스택은 아직 미정. 정해지면 아래에 포맷터·린터·테스트 명령을 추가한다.
> (예: TypeScript → prettier + eslint, Python → black + ruff, Go → gofmt + go vet)

- 언어가 정해지면 **공식/표준 포맷터를 그대로 쓴다.** 포맷 논쟁에 시간을 쓰지 않는다.
- 함수는 한 가지 일만 한다. 과한 추상화·이른 일반화를 피한다.
- 주석은 "왜"를 설명하고, "무엇"은 코드 자체로 드러나게 한다.
- 네이밍·구조는 그 언어/프로젝트의 관용을 먼저 따른다 (새 패턴을 함부로 도입하지 않는다).
- 죽은 코드·미사용 import 는 남기지 않는다.

## 9. 하지 말아야 할 것 (Do NOT)

- 요청하지 않은 대규모 리팩토링을 하지 않는다.
- 파일을 지우거나 덮어쓰기 전에 내용을 확인하고, 예상과 다르면 먼저 알린다.
- 비밀키·토큰을 코드나 커밋에 넣지 않는다.

## 10. 실행 진입점 (Entry Point)

- **프로젝트 실행은 항상 `./run.sh` 하나로만 한다.**
- `run.sh` 에는 환경 설정(env 로드, 의존성 설치, 경로 설정 등) + 실행 명령을 모두 포함한다.
- `run.sh` 하나만 실행해도 프로그램이 정상 동작해야 한다. 다른 사전 단계가 필요하면 그것도 `run.sh` 안에 넣는다.
- `.env` 파일이 필요하면 `.env.example` 을 먼저 확인하고, 없으면 `run.sh` 안에 필수 변수를 주석으로 명시한다.
- `run.sh` 를 수정하거나 새로 만들 때는 반드시 실행 권한을 확인한다: `chmod +x run.sh`

## 11. 자주 쓰는 명령 / 도구 (Cheatsheet)

- 실행: `./run.sh`
- 새 프로젝트 킥오프: `/start-project` 스킬 (CLAUDE.md 배치 + 추적 파일 생성 + 브레인스토밍 시작)
- 새 프로젝트 추적 파일만 세팅: `./scripts/init-project.sh` (현재 폴더에 양식 복사)
- 스킬 패키지 동기화: `./scripts/sync-skill.sh` (CLAUDE.md/templates 수정 후)
- 테스트/빌드: 스택 확정 후 `run.sh` 에 통합 (`./run.sh test` 등으로 확장)
- 자주 쓰는 스킬: `/deep-research`(조사), `/code-review`(PR 점검), `/verify`(동작 확인), `/security-review`(배포 전 보안)

## 12. 작업 완료 체크리스트 (Definition of Done)

- [ ] 요구사항을 모두 만족했는가
- [ ] 테스트/린트를 통과했는가
- [ ] **잘못된 입력·예외 상황 처리가 적용됐는가** (입력 검증, 모든 분기, 예외 미삼킴)
- [ ] **별도 에이전트의 검토·검증을 거쳤는가** (`/code-review`, `/verify`)
- [ ] 변경 범위가 의도한 것에만 한정됐는가
- [ ] 진행 추적 파일(plan / todo / work / history)을 갱신했는가
- [ ] 커밋 메시지가 변경 내용을 잘 설명하는가
