## 2. 작업을 시작하기 전 (Before starting)

- 코드를 바꾸기 전에 관련 파일·맥락을 먼저 파악한다.
- 요구사항이 모호하면 임의로 정하지 말고 선택지를 제시한다.
- 큰 작업은 먼저 계획을 세우고 합의한 뒤 진행한다.

## 3. 프로젝트 진행 관리 (Project Tracking)

> 모든 프로젝트는 **시작 단계에서 아래 추적 파일들을 먼저 만들고**, 진행하면서 계속 갱신한다.
> 역할이 겹치지 않게 파일을 분리한다.

### 시작 단계 (킥오프)
- 전체 계획과 **세부 주간 계획**을 먼저 작성한다.
- 작성한 계획은 `plan.md` 에 저장한다.
- 동시에 `todo.md`, `work.md`, `history.md`, `docs/` 를 함께 만든다.

### 추적 파일 구조
| 파일 | 역할 | 갱신 주기 |
|---|---|---|
| `plan.md` | 전체 계획 + 주간 세부 계획 + 현재 상태 요약 | 계획 변경 시 / 주간 |
| `todo.md` | 진행 상태 **대시보드** — 계획 대비 빠른지/늦은지, 현재 어느 구간(phase)인지 | 수시 |
| `work.md` | 할 일 / 한 일 체크리스트 (실제 작업 단위) | 작업할 때마다 |
| `history.md` | 매주 한 일을 간략히 기록 (주간 아카이브) | 주 1회 |
| `docs/` | 진행 중 발생하는 산출물 보관 | 산출물 생길 때마다 |

### 각 파일 규칙
- **plan.md** — 프로젝트 시작 시 작성. `전체 목표 → 마일스톤 → 주간 계획` 순으로. 맨 위에 "현재 상태" 요약을 둔다.
- **todo.md** — `계획 대비 +N일 빠름 / -N일 늦음`, `현재 구간: N주차 / 마일스톤 X` 처럼 진척 상태를 한눈에 보이게 한다. (거시 관점)
- **work.md** — 지금 할 일과 끝낸 일을 체크박스(`- [ ]` / `- [x]`)로 관리하고, 진행하면서 바로바로 반영한다. (미시 관점)
- **history.md** — 매주 끝에 그 주에 한 일을 3~5줄로 요약해 누적한다. 과거 기록은 지우지 않는다.
- **docs/** — 산출물은 `docs/YYYY-MM-DD_제목.md` 형식으로 저장한다 (날짜를 앞에 두면 시간순 정렬됨).

### 공통 원칙
- 각 파일 상단에 **마지막 업데이트 날짜**를 적는다.
- 작업을 끝낼 때 관련 추적 파일을 갱신하는 것을 "완료"의 일부로 본다 (10번 체크리스트 참고).

## 4. 일하는 방식 (How I work)

- 변경은 작고 검증 가능한 단위로 나눈다.
- 기존 코드의 스타일·네이밍·구조를 먼저 따른다 (새 패턴을 함부로 도입하지 않는다).
- 가능한 한 테스트로 검증한 뒤 "됐다"고 말한다. 안 됐으면 그대로 보고한다.

## 8. 실행 진입점 (Entry Point)

- **프로젝트 실행은 항상 `./run.sh` 하나로만 한다.**
- `run.sh` 에는 환경 설정(env 로드, 의존성 설치, 경로 설정 등) + 실행 명령을 모두 포함한다.
- `run.sh` 하나만 실행해도 프로그램이 정상 동작해야 한다. 다른 사전 단계가 필요하면 그것도 `run.sh` 안에 넣는다.
- `.env` 파일이 필요하면 `.env.example` 을 먼저 확인하고, 없으면 `run.sh` 안에 필수 변수를 주석으로 명시한다.
- `run.sh` 를 수정하거나 새로 만들 때는 반드시 실행 권한을 확인한다: `chmod +x run.sh`
